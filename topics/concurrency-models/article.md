# 并发编程模型

## 概念解释

并发编程模型是处理多个任务同时执行的方法和架构。

想象餐厅：
- **多进程**：多个厨师各自做菜
- **多线程**：一个厨师同时照看多个锅
- **协程**：厨师在等水开时去切菜
- **异步IO**：下单后去做别的，菜好了服务员通知

## 为什么重要

选择合适的并发模型：
- 提高程序响应速度
- 充分利用多核CPU
- 提高系统吞吐量
- 简化编程复杂度

## 并发 vs 并行

**并发（Concurrency）**：
- 多个任务交替执行
- 单核CPU也能实现
- 逻辑上的同时执行

**并行（Parallelism）**：
- 多个任务真正同时执行
- 需要多核CPU
- 物理上的同时执行

## 线程池

### 什么是线程池？

线程池预先创建一组线程，任务来了直接分配线程执行，不需要频繁创建和销毁线程。

### 为什么用线程池？

- **减少开销**：创建线程有开销
- **控制并发**：限制同时执行的线程数
- **任务排队**：线程忙时任务排队等待
- **资源管理**：统一管理系统线程

### 线程池参数

- **核心线程数**：始终存活的线程数
- **最大线程数**：线程池最大容量
- **任务队列**：存放等待执行的任务
- **拒绝策略**：队列满时的处理方式

### 使用场景

- Web服务器处理请求
- 批量任务并行处理
- 异步任务执行

## 协程

### 什么是协程？

协程是用户态的轻量级线程，由程序自己调度。

### 协程 vs 线程

| 特性 | 线程 | 协程 |
|------|------|------|
| 调度 | 操作系统调度 | 程序自己调度 |
| 切换开销 | 大（内核态切换） | 小（用户态切换） |
| 内存占用 | MB级 | KB级 |
| 数量限制 | 百级 | 万级 |
| 并行能力 | 可以真并行 | 单线程内并发 |

### 协程的优点

- **轻量**：可以创建大量协程
- **高效**：切换开销小
- **简单**：用同步方式写异步代码

### 协程的工作原理

协程在遇到IO操作时主动让出控制权，IO完成后恢复执行。

```
协程A: 读文件 → 让出 → ... → IO完成 → 继续
协程B:         → 执行 → 让出 →        → 执行
```

## 异步IO

### 什么是异步IO？

发起IO请求后不等待结果，继续做其他事情，IO完成后通过回调或通知获取结果。

### 同步 vs 异步

**同步IO**：
```
发起请求 → 等待 → 获取结果 → 继续
```

**异步IO**：
```
发起请求 → 继续做其他事 → IO完成通知 → 处理结果
```

### 异步编程模型

**回调函数**：
```javascript
readFile('data.txt', function(data) {
    console.log(data)
})
```

**Promise/Future**：
```javascript
readFile('data.txt')
    .then(data => console.log(data))
```

**async/await**：
```javascript
const data = await readFile('data.txt')
console.log(data)
```

## 事件驱动模型

### 什么是事件驱动？

程序的执行由事件决定，而不是顺序执行。

### 事件循环

1. 等待事件发生
2. 事件发生时调用对应的处理函数
3. 处理完成后继续等待

### 使用场景

- GUI程序（按钮点击、鼠标移动）
- Web服务器（处理HTTP请求）
- 游戏引擎（处理用户输入）

## Actor模型

### 什么是Actor？

Actor是并发计算的基本单元，每个Actor有自己的状态和邮箱。

### Actor的特点

- **独立**：Actor之间不共享状态
- **消息传递**：通过消息通信
- **并发**：Actor可以并行执行

### 使用场景

- 分布式系统
- 高并发服务器
- 游戏服务器

## CSP模型

### 什么是CSP？

CSP（通信顺序进程）通过通道（Channel）进行通信。

### 特点

- **通道**：用于goroutine之间通信
- **阻塞**：发送和接收都会阻塞
- **同步**：保证数据安全

### Go语言的实现

```go
ch := make(chan int)
go func() { ch <- 42 }()
value := <-ch
```

## 常见错误

**错误1：线程越多越好**

线程太多会导致上下文切换开销大，通常线程数设为CPU核心数或2倍。

**错误2：协程可以替代线程**

协程适合IO密集型任务，CPU密集型任务还是需要多线程。

**错误3：异步一定比同步快**

异步在IO等待多时有优势，纯计算任务异步没有优势。

## 实际应用

**Nginx**：事件驱动模型，高并发Web服务器。

**Node.js**：单线程事件循环，适合IO密集型。

**Go**：Goroutine + Channel，简单高效的并发。

**Java**：线程池 + CompletableFuture。

**Python**：asyncio + 协程。

## 总结

- 并发模型决定程序如何处理多任务
- 线程池减少线程创建销毁开销
- 协程是轻量级的用户态线程
- 异步IO提高IO密集型程序的性能
- 事件驱动适合处理大量IO事件
- 选择合适的模型取决于具体场景
