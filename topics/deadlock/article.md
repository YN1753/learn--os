# 死锁

## 概念解释

**死锁（Deadlock）** 是指两个或多个进程互相等待对方释放资源，导致所有进程都无法继续执行的情况。

想象一个场景：
- 你拿着筷子等碗
- 朋友拿着碗等筷子
- 你们谁也不肯放手
- 结果谁都吃不了饭

这就是死锁。

## 为什么重要

死锁会导致：
- 程序卡死，无法响应
- 资源被浪费，无法回收
- 系统性能严重下降
- 严重时需要重启系统

理解死锁能帮你：
- 写出不会死锁的程序
- 诊断系统卡死的原因
- 设计更可靠的并发系统

## 死锁的四个必要条件

死锁发生时，以下四个条件**同时成立**：

### 1. 互斥条件（Mutual Exclusion）

资源一次只能被一个进程使用。比如打印机一次只能打印一个文档。

### 2. 请求和保持条件（Hold and Wait）

进程已经持有至少一个资源，又在等待其他资源。就像你拿着筷子不放，还要等碗。

### 3. 不可抢占条件（No Preemption）

资源只能由持有它的进程主动释放，不能被强行夺走。就像不能从你手里硬抢筷子。

### 4. 循环等待条件（Circular Wait）

存在一个进程的循环链，每个进程都在等待下一个进程持有的资源。

```
进程A → 等待资源1 → 进程B → 等待资源2 → 进程C → 等待资源3 → 进程A
```

## 死锁的例子

### 银行转账问题

```
账户A有100元，账户B有200元

线程1：从A转50到B
  1. 锁定A
  2. 锁定B
  3. 执行转账
  4. 释放B
  5. 释放A

线程2：从B转30到A
  1. 锁定B
  2. 锁定A
  3. 执行转账
  4. 释放A
  5. 释放B
```

如果线程1锁定了A，同时线程2锁定了B，然后：
- 线程1等线程2释放B
- 线程2等线程1释放A
- 死锁！

## 处理死锁的策略

### 1. 死锁预防

破坏死锁的四个条件之一：

**破坏互斥条件**
- 使用可共享的资源
- 但有些资源天生就是互斥的（如打印机）

**破坏请求和保持条件**
- 要求进程一次性申请所有资源
- 可能导致资源浪费和饥饿

**破坏不可抢占条件**
- 允许系统强行夺走资源
- 实现复杂，可能影响程序正确性

**破坏循环等待条件**
- 给资源编号，要求按编号顺序申请
- 最实用的方法

### 2. 死锁避免

在分配资源前检查是否会导致死锁。

**银行家算法**：

模拟分配后检查系统是否处于**安全状态**——存在一个安全序列，能让所有进程顺利完成。

安全状态：能找到一个序列，使每个进程都能获得所需资源并完成。

不安全状态：可能存在死锁，但不一定发生。

### 3. 死锁检测

允许死锁发生，但定期检测并处理。

**资源分配图**：
- 用图表示进程和资源的关系
- 检测图中是否有环
- 有环不一定死锁，但需要进一步检查

### 4. 死锁恢复

检测到死锁后：

**终止进程**
- 终止所有死锁进程（简单粗暴）
- 逐个终止死锁进程直到死锁解除（需要选择"牺牲者"）

**资源抢占**
- 从某些进程强制收回资源
- 需要考虑回滚和饥饿问题

## 常见错误

**错误1：死锁只会发生在多线程程序中**

死锁可以发生在任何并发系统中，包括数据库事务、分布式系统、甚至操作系统本身。

**错误2：加锁就能解决并发问题**

不当的加锁策略反而会导致死锁。关键是**按顺序**获取锁。

**错误3：死锁检测可以解决所有问题**

检测到死锁后，终止进程或回滚操作都有代价，最好的策略是预防。

## 实际应用

**数据库系统**：使用超时机制和死锁检测来处理事务死锁。

**操作系统**：Linux 内核使用锁顺序来预防死锁。

**Java**：可以使用 `tryLock` 设置超时，避免无限等待。

**Python**：`threading` 模块提供 `Lock`、`RLock`、`Semaphore` 等同步原语。

## 如何避免死锁

1. **按固定顺序获取锁**：所有线程都按相同的顺序获取锁
2. **设置超时**：获取锁时设置超时，超时后释放已持有的锁
3. **减少锁的范围**：只在必要时持有锁
4. **使用无锁数据结构**：如 CAS 操作、原子变量
5. **避免嵌套锁**：尽量不要同时持有多个锁

## 总结

- 死锁是多个进程互相等待的僵局
- 四个必要条件：互斥、请求和保持、不可抢占、循环等待
- 处理策略：预防、避免、检测、恢复
- 最实用的方法：按顺序获取锁 + 设置超时
- 死锁是并发编程中必须面对的重要问题
