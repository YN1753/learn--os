# 同步与互斥

## 概念解释

**同步**：多个进程按约定的顺序执行。
**互斥**：同一时间只有一个进程能访问共享资源。

想象一个场景：
- 多个人（进程）要使用同一个打印机（共享资源）
- 打印机一次只能一个人用 → **互斥**
- 你先排队，我后排队 → **同步**

## 为什么重要

在并发环境中，如果不对共享资源进行保护：
- 数据可能被破坏（竞态条件）
- 结果可能不确定
- 程序可能崩溃

同步和互斥机制确保并发程序的正确性。

## 临界区问题

### 什么是临界区？

临界区是访问共享资源的代码段。同一时间只能有一个进程在临界区内执行。

### 临界区的三个部分

```
进入区 → 临界区 → 退出区
   ↓         ↓         ↓
 请求访问   访问资源   释放访问
```

### 临界区管理的原则

1. **互斥**：同一时间只有一个进程在临界区
2. **前进**：如果临界区空闲，等待进入的进程能在有限时间内进入
3. **有限等待**：每个进程等待进入临界区的时间是有限的

## 互斥的实现方法

### 1. 软件方法

**Peterson算法**（两个进程）：

```c
// 进程0和进程1共享
bool flag[2] = {false, false};
int turn = 0;

// 进程i的代码
flag[i] = true;       // 我想进入
turn = 1 - i;         // 让给对方
while (flag[1-i] && turn == 1-i) ; // 等待
// 临界区
flag[i] = false;      // 离开
```

**特点**：正确但复杂，不实用。

### 2. 硬件方法

**禁用中断**：
- 进入临界区前关中断
- 离开后开中断
- 简单但危险，不适合多核

**原子指令**：
- `TestAndSet`：原子地读取并设置变量
- `Swap`：原子地交换两个变量
- 硬件支持，高效

```c
// TestAndSet
bool TestAndSet(bool *target) {
    bool old = *target;
    *target = true;
    return old;
}

// 使用
bool lock = false;
while (TestAndSet(&lock)) ; // 忙等待
// 临界区
lock = false;
```

### 3. 互斥锁（Mutex）

最常见的互斥机制：

```c
mutex_lock(&lock);
// 临界区
mutex_unlock(&lock);
```

**特点**：
- 简单易用
- 获取不到锁时进程会阻塞
- 不会忙等待浪费CPU

### 4. 信号量（Semaphore）

由Dijkstra提出，比互斥锁更灵活：

```c
// P操作（等待）
wait(S) {
    while (S <= 0) ; // 忙等待
    S--;
}

// V操作（释放）
signal(S) {
    S++;
}
```

**类型**：
- **二值信号量**：0或1，类似互斥锁
- **计数信号量**：可以是任意非负整数

**特点**：
- 可以实现互斥
- 可以实现同步
- 可以控制资源数量

## 经典同步问题

### 1. 生产者-消费者问题

**场景**：
- 生产者往缓冲区放数据
- 消费者从缓冲区取数据
- 缓冲区满时生产者等待
- 缓冲区空时消费者等待

**信号量方案**：
- `mutex`：互斥访问缓冲区
- `empty`：空缓冲区数量
- `full`：满缓冲区数量

```
生产者:
P(empty)    // 等待空缓冲区
P(mutex)    // 进入临界区
放数据
V(mutex)    // 离开临界区
V(full)     // 增加满缓冲区

消费者:
P(full)     // 等待满缓冲区
P(mutex)    // 进入临界区
取数据
V(mutex)    // 离开临界区
V(empty)    // 增加空缓冲区
```

### 2. 读者-写者问题

**场景**：
- 多个读者可以同时读
- 写者必须独占访问
- 读者和写者不能同时访问

**方案**：
- `readcount`：当前读者数量
- `mutex`：保护readcount
- `wrt`：控制写者访问

```
读者:
P(mutex)
readcount++
if (readcount == 1) P(wrt)  // 第一个读者锁住
V(mutex)
读数据
P(mutex)
readcount--
if (readcount == 0) V(wrt)  // 最后一个读者释放
V(mutex)

写者:
P(wrt)
写数据
V(wrt)
```

### 3. 哲学家就餐问题

**场景**：
- 5个哲学家围坐圆桌
- 每人左右各有一根筷子
- 需要同时拿到两根筷子才能吃饭

**问题**：可能死锁——每人拿起左边的筷子，等右边的筷子。

**解决方案**：
1. 最多允许4人同时拿筷子
2. 拿筷子前先检查两边都可用
3. 给筷子编号，按编号顺序拿

## 管程

### 什么是管程？

管程是一种高级同步机制，将共享数据和操作封装在一起。

**特点**：
- 一次只有一个进程在管程内
- 管程内部可以有等待队列
- 比信号量更容易使用

### 管程的组成

1. **共享数据**：需要保护的变量
2. **操作过程**：操作共享数据的函数
3. **初始化代码**：初始化共享数据
4. **条件变量**：用于同步

## 常见错误

**错误1：忙等待可以接受**

忙等待浪费CPU资源，应该使用阻塞机制（如信号量的等待队列）。

**错误2：互斥就是同步**

互斥是排他访问，同步是顺序协调。互斥是同步的一种特殊情况。

**错误3：信号量初始值随意设置**

信号量的初始值有明确含义：互斥信号量初始为1，资源信号量初始为资源数量。

## 实际应用

**POSIX线程**：提供互斥锁（pthread_mutex）和条件变量（pthread_cond）。

**Java**：synchronized关键字和ReentrantLock类。

**数据库**：使用锁机制实现事务的隔离性。

**操作系统内核**：大量使用自旋锁、互斥锁保护内核数据结构。

## 总结

- 同步确保进程按正确顺序执行
- 互斥确保同一时间只有一个进程访问共享资源
- 实现方式：软件方法、硬件方法、互斥锁、信号量
- 经典问题：生产者-消费者、读者-写者、哲学家就餐
- 管程是更高级的同步机制
- 正确使用同步机制是并发编程的关键
