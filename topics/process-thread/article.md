# 进程与线程

## 概念解释

**进程（Process）** 是程序的一次执行。当你打开一个程序，操作系统就为它创建一个进程。

**线程（Thread）** 是进程中的执行单元。一个进程可以包含多个线程。

打个比方：
- 进程就像一个**工厂**，有独立的厂房、设备、原料
- 线程就像工厂里的**工人**，共享厂房和设备，各自干不同的活

## 为什么重要

理解进程和线程能帮你明白：
- 为什么多个程序能同时运行
- 为什么程序会卡死
- 为什么多线程能提高性能
- 为什么多线程编程容易出 bug

## 进程的状态

一个进程有以下状态：

1. **新建（New）**：进程正在被创建
2. **就绪（Ready）**：进程准备好运行，等待 CPU
3. **运行（Running）**：进程正在 CPU 上执行
4. **阻塞（Blocked）**：进程在等待某个事件（如读取文件）
5. **终止（Terminated）**：进程执行完毕

状态转换：
- 新建 → 就绪：进程创建完成
- 就绪 → 运行：被调度器选中
- 运行 → 就绪：时间片用完
- 运行 → 阻塞：等待 I/O
- 阻塞 → 就绪：I/O 完成
- 运行 → 终止：执行结束

## 进程 vs 线程

| 特性 | 进程 | 线程 |
|------|------|------|
| 内存空间 | 独立 | 共享 |
| 创建开销 | 大 | 小 |
| 通信方式 | IPC（管道、消息队列等） | 共享内存 |
| 崩溃影响 | 不影响其他进程 | 整个进程崩溃 |
| 资源占用 | 多 | 少 |

## 进程调度

操作系统需要决定哪个进程使用 CPU，这就是**调度**。

### 常见调度算法

**1. 先来先服务（FCFS）**
- 谁先来谁先执行
- 简单但可能导致"护航效应"——短进程被长进程阻塞

**2. 短作业优先（SJF）**
- 执行时间短的优先
- 平均等待时间最短
- 但可能饿死长进程

**3. 时间片轮转（RR）**
- 每个进程轮流执行一小段时间（时间片）
- 公平，适合交互式系统
- 时间片太小 → 上下文切换开销大
- 时间片太大 → 退化成 FCFS

**4. 优先级调度**
- 优先级高的先执行
- 可能导致低优先级进程饿死
- 解决方案：老化（逐渐提高等待进程的优先级）

## 上下文切换

当 CPU 从一个进程切换到另一个时，需要：
1. 保存当前进程的状态（寄存器、程序计数器等）
2. 加载新进程的状态
3. 开始执行新进程

这个过程叫**上下文切换**，它有开销，所以切换太频繁会降低性能。

## 多线程的优势

1. **响应性**：一个线程阻塞，其他线程继续工作
2. **资源共享**：线程共享进程的内存，通信更高效
3. **经济性**：创建线程比创建进程快得多
4. **多核利用**：多线程可以真正并行执行

## 常见错误

**错误1：进程和线程是一样的**

进程有独立内存空间，线程共享进程内存。进程间通信需要特殊机制，线程间可以直接读写共享变量。

**错误2：多线程一定比单线程快**

多线程有同步开销，如果任务是纯 CPU 计算且没有并行需求，单线程可能更快。

**错误3：线程越多性能越好**

线程太多会导致频繁的上下文切换，反而降低性能。

## 实际应用

**浏览器**：每个标签页通常是独立进程，一个标签页崩溃不会影响其他标签页。

**Web 服务器**：使用多线程同时处理多个用户请求。

**游戏**：一个线程处理渲染，一个线程处理物理计算，一个线程处理网络通信。

## 总结

- 进程是程序的执行实例，有独立内存空间
- 线程是进程内的执行单元，共享进程资源
- 调度算法决定哪个进程获得 CPU 时间
- 上下文切换是进程切换的必要开销
- 多线程提高并发性但增加编程复杂度
